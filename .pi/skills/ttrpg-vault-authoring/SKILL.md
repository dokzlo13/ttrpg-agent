# ttrpg-vault-authoring

The vault is user data, not repo structure. The active authored workspace is
`vault/notes/`, and its internal layout is intentionally flexible. Your job is
to keep it useful without forcing every note into a pre-baked taxonomy.

This skill contains high-level vault-organization policy: use YAML for filtering,
body wikilinks for relationships, and preserve hard data boundaries between
active notes, generated library books, and imports.

## Compose with focused vault skills

Research and Pi's progressive-disclosure model both favor small, focused skills
over one very large vault skill. Use this routing:

- Use **this skill** for destination, file boundaries, graph obligations,
  frontmatter policy, and whether to create stubs.
- Use **`ttrpg-vault-rich-notes`** when a Markdown note should be table-ready or
  Obsidian-rich: aliases, callouts, embeds, block IDs, detailed note patterns,
  rich section structure, or source/citation polish.
- Use **`ttrpg-vault-canvas`** when creating or editing `.canvas` files,
  relationship maps, clue boards, timelines, or visual session prep boards.
- Use **`ttrpg-system-qmd-maintenance`** after major note creation/migration when
  the searchable index should be refreshed.

Do not copy every rich-note detail here. Load the focused skill when needed.

## Hard boundaries

- `vault/notes/` is writable active campaign data.
- `vault/library/books/` is generated by book ingest; read/cite it, but do not hand-edit ingested chapters.
- `imports/` is input/reference data and should not be modified except when the user explicitly adds books, a legacy vault, or a 5etools clone.
- `.ttrpg/` is project machinery/runtime state, not a place for notes.
- `.canvas` files in the active vault should still live under `vault/notes/` unless the user requests another vault location.

## Placement principles

1. **Prefer existing local structure.** Before writing, inspect `vault/notes/` with
   `find vault/notes -maxdepth 2 -type d` and reuse nearby folders if they exist.
2. **Create obvious folders when useful.** Good examples under `vault/notes/`:
   `npcs/`, `locations/`, `sessions/`, `mechanics/monsters/`, `readalouds/`,
   `handouts/`, `images/`, `canvases/`. These are hints, not requirements.
3. **Avoid junk-drawer folders.** If placement is not obvious, choose the best
   semantic folder and mark `status: draft`. Ask one focused question only when
   placement changes meaning or no semantic destination is defensible.
4. **Do not hand-edit book ingests.** `vault/library/books/<book-slug>/` is owned
   by `ttrpg-import-book-pdf` / `book-ingest`. You may read and cite it, but re-ingest instead of editing.
5. **Keep all filenames slugged.** Kebab-case, ASCII, no spaces.
6. **Prefer stable hub locations.** If a note will become a recurring campaign
   hub, place it where future agents will naturally find it rather than in a
   temporary session folder.

## Link graph requirements

Every durable note should improve the Obsidian graph.

- Use wikilinks for important entities: `[[dunemark]]`, `[[lord-blackthorne|Lord Blackthorne]]`.
- Add a `## Connections` section for reusable notes with bullets like:
  `- [[dunemark]] — hires caravans from here.`
- If a link target does not exist and the target matters, create a short stub in
  an appropriate semantic folder under `vault/notes/`.
- Prefer body links over frontmatter-only relationships; Obsidian's graph is
  driven by links in note text.
- When linking from active notes to ingested book indexes, do **not** invent a
  slug note like `[[the-pit-in-the-forest]]` unless that note exists. The book
  index file is usually named `_book.md`, so use a path-qualified Obsidian link
  from the vault root, e.g. `[[library/books/the-pit-in-the-forest-v1-2-basic-bx/_book|The Pit in the Forest]]`.
  Section/chapter links may use their generated filenames when unique, e.g.
  `[[09-3-hollow-eyed-charcoal-burners|3 Hollow-eyed charcoal burners]]`.
- For generated artifacts, link back to the source/context note when known.
- For canvases, prefer file nodes pointing at existing notes and add a companion
  Markdown note if the canvas needs frontmatter, long sources, or qmd indexing.

## Minimal frontmatter

Use frontmatter for filtering, not as a substitute for links:

```yaml
---
type: npc | location | faction | session | monster | item | spell | rules | readaloud | handout | canvas | meta | draft
source: agent | user | imports/source-vault/<path> | imports/books/<file>.pdf | vault/library/books/<book>/<chapter>.md
created: YYYY-MM-DD
tags: [campaign]
status: draft | reviewed | canon
---
```

Optional but useful properties for richer notes:

```yaml
aliases: []
cssclasses: []
party_level: 5
session: "[[session-12]]"
```

Keep optional fields purposeful. Do not replace body links with YAML-only
relationships.

## Body template for most notes

```markdown
# Title

## Summary
One short paragraph.

## Details
Useful table-facing content.

## Connections
- [[related-note]] — why it matters.

## Sources
- agent
- source path or linked local chapter when applicable
```

For richer notes, load `ttrpg-vault-rich-notes` and use its patterns for
At-a-Glance sections, callouts, embeds, block IDs, secrets, clues, session prep,
and source polish.

## Rich note triage

Choose the lightest note shape that will remain useful:

- **Stub:** title, summary, one connection, sources. Use when creating a target
  only so links resolve.
- **Usable note:** summary, details, connections, sources. Default for most
  durable campaign facts.
- **Table-ready note:** add at-a-glance bullets, running cues, secrets/clues,
  callouts, and stable block IDs. Use for current-session material.
- **Hub note:** add child sections, embeds, index-like link lists, and possibly a
  companion canvas. Use for factions, locations, arcs, mysteries, or dungeons.

When a note reaches table-ready or hub status, read `ttrpg-vault-rich-notes`.

## Canvas authoring triage

Create a `.canvas` when visual layout adds value beyond headings or a Mermaid
snippet:

- relationship maps between NPCs/factions/locations
- mystery clue boards and revelation maps
- encounter/session flow diagrams with linked notes
- visual campaign or faction clocks

Default destination: `vault/notes/canvases/<slug>.canvas`.

Canvas files do not carry YAML. If the canvas is significant, create a companion
`vault/notes/canvases/<slug>.md` with frontmatter, summary, sources, and links to
major notes. Use `ttrpg-vault-canvas` for JSON structure and validation.

## Default routing hints

These are hints, not laws. Prefer an existing better local folder when one exists.

| Content | Preferred destination if no better local folder exists |
|---|---|
| uncertain draft | best semantic folder with `status: draft`; ask if placement changes meaning |
| NPC | `vault/notes/npcs/<slug>.md` |
| location | `vault/notes/locations/<slug>.md` |
| faction | `vault/notes/factions/<slug>.md` |
| session prep/recap | `vault/notes/sessions/<slug>.md` |
| monster/item/spell/rule | `vault/notes/mechanics/<kind>/<slug>.md` |
| read-aloud | `vault/notes/readalouds/<slug>.md` |
| handout | `vault/notes/handouts/<slug>.md` |
| generated image | `vault/notes/images/<slug>.<ext>` plus optional adjacent asset note |
| Obsidian canvas | `vault/notes/canvases/<slug>.canvas` plus optional companion `.md` |
| ingested book | `vault/library/books/<book-slug>/` via `ttrpg-import-book-pdf` / `book-ingest` only |

## Stubs

Create a stub when an important wikilink target does not exist and the target is
likely to recur. Keep stubs honest and small:

```markdown
---
type: npc
source: agent
created: YYYY-MM-DD
tags: [campaign]
status: draft
---

# Mara Vale

## Summary
Stub for a recurring smuggler captain mentioned in [[session-12]].

## Connections
- [[session-12]] — first mentioned here.

## Sources
- agent
```

Do not create stubs for throwaway names, one-off rumors, or generated book
chapter titles unless the user wants them promoted into campaign memory.

## After writing

- Tell the user the file path.
- Provide a short excerpt or summary of what was written.
- If the note should be discoverable through qmd immediately and the task scope
  includes index maintenance, run/update through `ttrpg-system-qmd-maintenance`.
- If you created a canvas, validate it with `ttrpg-vault-canvas` and report that
  validation passed.

## Don't

- Don't recreate the old numbered skeleton unless the user asks.
- Don't leave important isolated notes without `## Connections`.
- Don't rely on frontmatter alone for relationships.
- Don't write active notes outside `vault/notes/`.
- Don't write anything into `imports/`; it is input/reference data.
- Don't use a canvas to hide canon facts that should be searchable in Markdown.
